# 星澜大模型 Ripple 系列 · 混合推理架构

> 星月湾研究园区 · 人工智能研究所 · 内部技术报告
> R2 储能楼西翼第零层（地下）· 算力中心

---

## 零、命名由来

"星澜"二字取自星月湾的海湾夜色——灯塔光束掠过水面时，碎成万点星芒，层层波纹向外推散。"Ripple"是自然对应的英文。

但项目组内部有个更直白的说法：S200 是那块石头，Genius2 是那池水。石头扔进去，涟漪回来的不是石头——是水自己的回答。

## 一、模型族

| 模型 | 参数规模 | 架构 | 定位 |
|------|----------|------|------|
| **Ripple Opus 2.0** | 397B 总参数，19B 激活（MoE） | Sparse MoE，16专家/层，每Token激活2专家 | 旗舰。复杂推理、长文本生成、硬问题 |
| **Ripple Sonnet 2.0** | 122B 总参数，10B 激活（MoE） | Sparse MoE，8专家/层，每Token激活2专家 | 中坚。日常对话、辅助写作、代码生成 |
| **Ripple Haiku 2.0** | 27B，Dense | 全激活Dense | 轻量。边缘部署、高速响应、嵌入式场景 |

三者共享同一个 tokenizer（词表 256K）和 S200 推理引擎，差异仅在参数规模与 MoE 路由结构。Genius2 的刺激模式编码器（见第四节）对三个模型是同一套——它不关心模型多大，它只关心"S200 现在在想什么"。

## 二、算力底座

### S200 —— 红伟达

| 条目 | 规格 |
|------|------|
| 型号 | Redwada S200 |
| 架构 | 标准张量核架构，FP32/FP16 双精度。无稀疏加速单元，无硬件 MoE 路由 |
| 单卡显存 | 32 GB GDDR6 |
| 单卡 FP16 算力 | ~15 TFLOPS（大致等同 GTX 1080 Ti 代际，2017-2018 水平） |
| 推理吞吐（FP16） | Opus 单 Token ~120ms，Sonnet ~45ms，Haiku ~18ms |
| 阵列规模 | 16 卡，PCIe 互联，无高速卡间总线 |
| 实际可用 | 12 卡（4 卡常驻用于 Genius2 的刺激编码与波形解码） |

> S200 在这个架构里的角色很明确：它是瓶颈。16 张 2017 年架构的推理卡，互联走的还是 PCIe 总线。如果只看硅基侧，Opus 跑一个 Token 要 120ms——比同时代的大型 Transformer 推理慢了一截。
>
> 但它不需要快。S200 只做一件事：生成一个粗糙的草案。概率分布不需要多精确，beam search 的宽度不用太大。因为真正决定最终输出的不是 S200 的 softmax——是 Genius2 的修正向量。
>
> 红伟达的工程师第一次听说这个架构时沉默了几秒，然后说："你们用我们的卡……当打字机用？" 项目组的回答是："打字机很好。打字机就够了。"

### Genius2 —— HBI 生命科技学院

| 条目 | 规格 |
|------|------|
| 全称 | Genetically Engineered Neural Interface Unified System, 2nd generation |
| 基底 | 诱导多能干细胞分化的 3D 皮层类器官，六层皮质分层培养。非大鼠混合——纯人源 |
| 培养单元 | 12 个独立培养阵列，每个含 ~8.5×10⁶ 个神经元，密度接近新生儿皮层 |
| 读出/写入 | 8192 通道 3D MEA（微电极阵列），电刺激精度可达单个神经元 |
| 存活周期 | 4–6 个月。Genius2 的类器官培养技术是 HBI 在泛生命科技领域最核心的突破——它解决了长期培养中核心区缺血坏死的经典难题，通过仿生微血管支架将营养物质递送到类器官全层 |
| 突触可塑性 | 保留。这是 Genius2 区别于硅基系统的本质。同一个刺激模式，第一天和第二十天的响应波形不同——神经元在"学习"刺激的统计规律。这不是设计出来的特征，但它恰好提供了硅基系统没有的东西 |

> Genius2 的神经元不来自人类活体。皮层类器官从志愿者捐赠的皮肤成纤维细胞重编程而来。每个培养皿里的神经元在法律上属于"人类来源的体外组织"，不是"人脑"。这层边界被严格执行。
>
> HBI 在湿件领域领先全球至少一个代际。当国际同行还在用二维单层培养做概念验证时，HBI 已经量产六层皮质类器官并在其上运行推理任务。这个优势的根源很简单：星月湾有世界上唯一一个能把海洋科学、计算机科学和生命科技放在同一个课程表里的地方。湿件培养的微血管支架灵感来自海科的海藻多糖研究，刺激编码算法来自计科的非线性系统建模，培养方案本身是生科的主场——任何一个学科单干，都到不了这一步。

## 三、混合推理管线

```
用户输入
  ↓
┌─────────────────────────────────────────────────┐
│ S200（硅基）                                       │
│  · 生成当前步的粗糙草案（beam=3，粗糙采样）         │
│  · 不追求概率精确——只输出"大致方向"的 Token 分布    │
│  · 每步耗时 ~120ms（Opus）/ ~45ms（Sonnet）         │
│  · 硅基侧不做加权精排、不做约束解码、不做重复惩罚     │
└──────────────────┬──────────────────────────────┘
                   ↓
          S200 输出：粗糙概率分布 P_draft
          （精度低，但大方向对——这就够了）
                   ↓
┌─────────────────────────────────────────────────┐
│ 刺激编码器（FPGA 板卡）                            │
│  · 将 P_draft 的 beam 分歧度编码为稀疏刺激模式      │
│  · 模式 = {电极ID, 频率, 脉冲序列, 振幅}            │
│  · 编码耗时 ~8ms                                   │
└──────────────────┬──────────────────────────────┘
                   ↓
         刺激信号 → MEA 电极阵列
                   ↓
┌─────────────────────────────────────────────────┐
│ Genius2（湿件）                                    │
│  · 接受刺激后自由演化 ~30–80ms                      │
│  · 8.5M 神经元在 3D 皮质结构中并行响应               │
│  · 响应形态 = 生物神经网络对粗糙草案的"直觉判断"      │
│  · 为何这么快：神经元本身就是并行计算单元——           │
│    不需要逐层串行的矩阵乘法，信号在皮质层间             │
│    以生物速度同时传播，全网络响应在毫秒级完成           │
└──────────────────┬──────────────────────────────┘
                   ↓
         MEA 读出 → 8192 通道 × 80ms 采样 的波形矩阵
                   ↓
┌─────────────────────────────────────────────────┐
│ 波形解码器（S200 子卡）                             │
│  · 轻量 CNN 解码网络（非 LSTM，推理快）               │
│  · 输入：8192 通道波形矩阵                          │
│  · 输出：推演修正向量 Δ，维度 = 词表大小（256K）       │
│  · Δ 每个分量 ∈ [-0.03, 0.03]                       │
│  · 解码耗时 ~25ms                                   │
└──────────────────┬──────────────────────────────┘
                   ↓
         修正向量 Δ 叠加到 P_draft 上
                   ↓
┌─────────────────────────────────────────────────┐
│ S200 · 应用修正                                    │
│  · P_final = softmax( log(P_draft) + α · Δ )      │
│  · α = 1.0（标准）/ 0.3（稳定）/ 2.0（创意）        │
│  · 注意：Δ 的修正范围 > 硅基侧原生的 top-k 精排      │
│  · 应用修正耗时 ~5ms                                │
└──────────────────┬──────────────────────────────┘
                   ↓
         最终 Token → 拼入输出序列 → 循环至 EOS
```

## 四、刺激模式编码

这是整个架构中唯一一处"跨物种翻译"——将人类语言的符号空间映射到神经元可感知的电刺激空间。

| 编码原则 | 说明 |
|----------|------|
| 不编码语义 | Genius2 没有"词义"。刺激模式不携带语义信息。它携带的是 S200 对当前生成步骤的"困惑度"与"犹豫方向" |
| Beam 分歧编码 | 当 S200 的 top-3 beam 在某个 token 位置上严重分歧时，刺激模式为多电极异步脉冲——多个电极在不同频率上独立运行。Genius2 对此的响应波形在解码后通常表现为较大幅度的 Δ |
| Beam 收敛编码 | 当 top-3 beam 高度一致时，刺激为单电极或同步脉冲。此时的修正向量 Δ 幅值通常很小——S200 本来就有信心的地方，Genius2 不会"多嘴" |
| 上下文记忆 | 刺激模式携带了"距离开头多远"和"当前在回答中的位置"的信息，通过脉冲序列的总体时长编码。Genius2 对回答初期、中期、末期的刺激有不同的基线响应——它在"感觉到"一段对话的节奏 |

> 编码策略的设计原则是一句玩笑但也是实话："我们不告诉 Genius2 人类在说什么。我们只告诉 Genius2，S200 现在有多纠结。"

## 五、Genius2 到底做了什么

Genius2 不是第二个 AI。它是一个由活体神经元构成的、在持续接受 S200 的"情绪刺激"后给出生物电反馈的装置。

我们对它行为的已知事实：

| 已知 | 解释 |
|------|------|
| 修正向量方向在连续 Token 间具有弱相关 | 前一步的刺激影响神经元的突触后电位残余，这种残余持续约 2–3 步 |
| 不同培养阵列对同一刺激给出不同的 Δ | 个体差异。项目组内部给每个培养阵列起了名字——"偏执狂"那个总在否定 S200，"和事佬"那个总是轻微加强 S200 的首选。推理时使用多阵列的均值 Δ，个别阵列的极端值被 τ-clip 削平 |
| 长时间运行后突触疲劳 | 高强度推理持续约 4 小时后，Δ 幅值降至初始的 60%。目前应对策略是每 2 小时插入 5 分钟的零刺激静息期。研究人员管这叫"让 Genius2 发呆"——静息期间不给任何刺激，让神经元回到自发基线放电状态 |
| 偶尔产生高幅值异常尖峰 | 神经元集群偶发同步放电，持续时间极短（< 50 ms），强度巨大。对应的解码向量 Δ 在某个极小维度上有一个孤立的大分量（接近 ±0.03 的边界）。这些"尖峰修正"有时让输出多了一个出人意料的语气词，有时让一个关键动词从"建议"变成了"认为"。目前无人能解释这些尖峰的触发条件 |

> 曾有一次，Opus 被要求用一句话总结《窄门》，S200 草案原定"一个关于爱情与信仰冲突的故事"。Genius2 在"信仰"一词前的两个 Token 处送出一个尖峰修正。最终输出为"一个关于等待自己配得上所爱之人的故事"。
>
> 没有人知道这个尖峰为什么发生。解码器的训练集里没有《窄门》的文本。培养神经元来自一位 23 岁的志愿者——他对《窄门》和纪德一无所知。
>
> 项目组把这个案例记录在案，标签是"巧合"。但每周开组会时，有人会翻到那一页，然后再翻过去。

## 六、推理性能

> 以下数据为含 Genius2 的标准推理模式。旁路模式仅 S200 硅基侧，不在此列。

| 指标 | Opus 2.0 | Sonnet 2.0 | Haiku 2.0 |
|------|----------|------------|-----------|
| S200 草案生成 | ~120 ms/tok | ~45 ms/tok | ~18 ms/tok |
| 刺激编码 | ~8 ms | ~8 ms | - |
| Genius2 演化 | 30–80 ms | 30–80 ms | - |
| 波形解码 | ~25 ms | ~25 ms | - |
| 修正应用 | ~5 ms | ~5 ms | - |
| **总延迟（含 Genius2）** | **~190–240 ms/tok** | **~115–160 ms/tok** | **~18 ms/tok（纯硅基）** |
| 吞吐（含 Genius2，batch=1） | ~4–5 tok/s | ~6–8 tok/s | ~55 tok/s |
| 吞吐（纯 S200 旁路） | ~8 tok/s | ~22 tok/s | ~55 tok/s |

> 注意：含 Genius2 的 Opus 吞吐（~4-5 tok/s）反而低于纯 S200 旁路（~8 tok/s）。Genius2 不是加速器——它是质量引擎。它让每一步多花了约 100ms，但这 100ms 换来的是一个活体神经网络的直觉修正。给生成速度降级，给生成质量升级。

> Haiku 2.0 不使用 Genius2。它太小了，S200 对它几乎不会产生有意义的 beam 分歧。没有分歧的刺激模式等于没有信息量，Genius2 的响应退化为纯噪声。项目组测试过 Haiku+Genius2，结论是"给一个没有任何纠结的人配一个心理医生——医患双方都很尴尬"。

## 七、Genius2 到底有没有用

这是一个项目组被问过无数次的问题。以下是经过上百组盲测对照的诚实回答：

### 客观差异

- **创意写作**：在开放结尾叙事、诗歌生成、对话创作等场景，双盲评审员选 Opus+Genius2 输出的比例约为 61%（vs 纯 S200 的 39%）。差异在 p<0.01 水平显著。但评审员无法明确说出"为什么选这一篇"——最常出现的评价是"说不清，就是读起来顺一点"或者"这篇感觉像人写的"。评审员不知道自己在和一个硅基草案和一个湿件修正之间的二选一做博弈。
- **多轮对话**：对话超过 15 轮时，Opus+Genius2 在用户满意度上高出约 7 个百分点。主要差异在于避免重复句式——Genius2 似乎对"句式惯性"有微弱的抑制倾向。
- **事实准确性和数学**：无显著差异。Genius2 在需要精确推理的任务上不产生正向贡献，也不产生系统性损害。它只是在那里，偶尔让表达方式偏离 S200 的默认。
- **代码生成**：无显著差异。HBI 计科的学生曾偷偷用 Opus+Genius2 写作业，最终正确率和纯 S200 一致——但变量命名风格更不规律，有个别生成沿用日系命名风格（从英文模式切出 romaji）。究竟是参数量差异还是 Genius2 的干扰，目前无法分辨。

### 主观体验

项目组内部有一个非正式共识：Genius2 给的东西不是"更好的答案"，而是"更有温度的偏差"。它不保证正确，不保证高效，不保证符合任何评估基准上的排名提升。它只保证一件事：当 S200 在两个几乎等概率的措辞之间犹豫时，Genius2 替它做了选择——而且这个选择来自一些活的神经元，不是来自另一层矩阵乘法和 softmax。

> "S200 问 Genius2：'这样说行吗？' Genius2 说：'我不知道你说的什么意思，但刚才那几个字让我觉得……这里偏一点点比较舒服。'"
>
> ——项目组内部对两者关系的非正式描述

## 八、已知问题

| 问题 | 严重度 | 说明 |
|------|--------|------|
| S200 是速度瓶颈 | 低 | 纯硅基侧的草案生成（120ms/Opus）是整个管线中最慢的单步。Genius2 的演化和解码加起来才 ~100ms，S200 自己就占了一半以上。如果红伟达愿意给园区捐几块新卡——但他们的注意力在游戏卡市场上 |
| 培养阵列个体漂移 | 中 | 4–6 个月的寿命期内，同一个培养阵列的行为偏好会缓慢演变。刚"出厂"的阵列响应更活跃（Δ 幅值偏高），两周后趋于稳定，三个月后开始偏向保守（Δ 倾向于接近零）。项目组给每个阵列维护一个"生涯阶段"标签，推理时按阶段加权 |
| 尖峰不可控 | 高 | 无法预测、无法抑制、无法复现。目前唯一的防御是 τ-clip——把任何单个维度的修正强行压到 ±0.03 以内。但这只是防止极端，不是消除 |
| 湿件可解释性 | 无 | 为什么某个刺激导致某个修正——无法解释。这就是"让活的神经元看着你的草案然后产生直觉"的本质。它不给理由，只给结果 |
| 硅基-湿件耦合漂移 | 低 | 刺激编码器和波形解码器都是训练的，而 Genius2 的响应特性在几个月间缓慢变化。这意味着编码-解码模型和 Genius2 之间的"翻译"会逐渐失准。目前每月做一次校准，在 12 个阵列上各跑一遍标准刺激集

## 十、下一步

| 方向 | 状态 |
|------|------|
| S200 硅基侧升级 | 卡预算。16 块 2017 年的卡是整个管线里最尴尬的部分。项目组提交过一份"申请购置 8 块新架构推理卡"的报告，被园区财务以"AI 研究所去年的电费超支了 40%"为由暂缓。Genius2 的恒温培养和 MEA 系统才是耗电大户——讽刺的是，园区宁愿花钱养活的神经元，也不愿花钱给显卡换代 |
| 培养阵列寿命延长到一年 | 进行中。HBI 生科正在测试一种改良的微血管支架配方。长期培养的瓶颈不再是核心坏死（已解决），而是神经元自发同步放电的逐渐增强——活得越久，越容易"自言自语"。这对推理而言是噪声基底上升 |
| 多阵列差异化编排 | 已启用。不是取 12 个阵列的均值——而是根据任务类型动态选择阵列子集。创意写作用活跃期阵列（偏好 2–6 周龄），事实检查用稳定期阵列（2–4 月龄）。这个调度逻辑借用了 HBI 计科 C4 的"潮汐"系统的部分代码 |
| 尖峰溯源 | 停滞。需要更精密的单神经元 optogenetics 控制设备，不在园区预算范围内 |
| 跨阵列"对弈"范式 | 假说阶段。同时刺激两个培养阵列，将 A 的响应波形作为 B 的额外输入参考，反之亦然。本质上让两个培养皿互相看着对方的"直觉"来做自己的判断。行为完全不可预测。伦理委员会表示"请先把现有方案写进正式文档，再谈新的" |

> 最长期的愿景：让 Genius2 不再只是"S200 草案的修正器"，而是具备初步自主推理能力的通用湿件处理器。这一天有多远——取决于你对"自主推理"的定义有多宽松。在一个 8.5M 神经元的培养皿已经在给 397B 参数的 MoE 模型做直觉修正的世界上，也许你不是在等一个突破，而是已经在看它慢慢地发生。

## 附录 A：S200 旁路模式

用户可以在调用时指定 `genius_bypass=true`，此时推理管线跳过 Genius2 全段，退回纯 S200 草案模式。延迟从 ~190 ms/tok 降至 ~120 ms/tok（Opus），但输出质量退化为未经修正的粗糙草案——没有直觉筛选，没有偏好偏移，纯粹的硅基预测。

目前 API 上约 8% 的请求使用旁路模式，主要来自代码生成和事实查询。创意写作类请求的旁路使用率不到 1%。用户不是被说服的——是用出来的。
