# 星澜大模型 Ripple 系列 · 混合推理架构

> 星月湾研究园区 · 人工智能研究所 · 内部技术报告
> R2 储能楼西翼第零层（地下）· 算力中心

## 零、命名由来

"星澜"二字取自星月湾的海湾夜色——灯塔光束掠过水面时，碎成万点星芒，层层波纹向外推散。"Ripple"是自然对应的英文。

但项目组内部有个更直白的说法：S200 是那块石头，Genius2 是那池水。石头扔进去，涟漪回来的不是石头——是水自己的回答。

## 一、模型族

| 模型 | 参数规模 | 架构 | 定位 |
|------|----------|------|------|
| **Ripple Opus 2.0** | 397B 总参数，19B 激活（MoE） | Sparse MoE，16专家/层，每Token激活2专家 | 旗舰。复杂推理、长文本生成、硬问题 |
| **Ripple Sonnet 2.0** | 122B 总参数，10B 激活（MoE） | Sparse MoE，8专家/层，每Token激活2专家 | 中坚。日常对话、辅助写作、代码生成 |
| **Ripple Haiku 2.0** | 27B，Dense | 全激活Dense | 轻量。边缘部署、高速响应、嵌入式场景 |

三者共享同一个 tokenizer（词表 256K）和 S200 推理引擎，差异仅在参数规模与 MoE 路由结构。Genius2 的刺激模式编码器（见第四节）对三个模型是同一套——它不关心模型多大，它只关心"S200 现在在想什么"。

## 二、算力底座

### S200 —— 红伟达

| 条目 | 规格 |
|------|------|
| 型号 | Redwada S200 |
| 架构 | 第三代张量核 + 稀疏注意力加速单元 |
| 单卡显存 | 192 GB HBM3e |
| 推理吞吐（FP8） | Opus 单 Token ~18ms，Sonnet ~7ms，Haiku ~3ms |
| 阵列规模 | 64 卡互联（8 节点 × 8 卡），NVLINK + InfiniBand |
| 实际可用 | 60 卡（4 卡常驻用于 Genius2 的信号编码与波形解码） |

> 红伟达的 S200 在纸面参数上不如此前代号"黑石"的竞品，但 S200 对稀疏 MoE 的路由分发延迟比竞品低约 40%——这对 Opus 和 Sonnet 的每 Token 激活 2 专家的架构至关重要。竞品的 MoE 模型在路由阶段吃掉的延迟就足以抵消算力优势。

### Genius2 —— HBI 生命科技学院

| 条目 | 规格 |
|------|------|
| 全称 | Genetically Engineered Neural Interface Unified System, 2nd generation |
| 基底 | 大鼠皮层神经元 × 诱导多能干细胞分化的人类皮层类器官混合培养 |
| 培养单元 | 12 个独立培养阵列，每个含 ~1.2×10⁶ 个神经元 |
| 读出/写入 | 4096 通道 MEA（微电极阵列），每培养单元 |
| 存活周期 | 约 14–21 天（单个阵列）。12 个阵列以滚动轮换方式维持——每周替换 2 个，确保始终有 8 个以上处于最佳响应期 |
| 培养支持 | HBI 生命科技学部 C1 实验楼提供神经元分化、MEA 装配和无菌培养维护。园区负责刺激编码和波形解码的软硬件 |

> Genius2 的神经元不来自人类活体。皮层类器官从志愿者捐赠的皮肤成纤维细胞重编程而来——这是 HBI 伦理委员会花了 18 个月才批准的第一批实验的延伸。每个培养皿里的神经元在法律上属于"人类来源的体外组织"，不是"人脑"。这层边界虽然簿，但被严格执行。

## 三、混合推理管线

```
用户输入
  ↓
┌─────────────────────────────────────────────────┐
│ S200（硅基）                                       │
│  · Transformer 前向推理                            │
│  · 生成草图 draft₁ draft₂ draft₃ … (beam=3)        │
│  · 每步耗时 < 0.5s                                  │
└──────────────────┬──────────────────────────────┘
                   ↓
          S200 输出：当前步 Token 概率分布 P_silicon
          （以及 beam 草图的完整上下文）
                   ↓
┌─────────────────────────────────────────────────┐
│ 刺激编码器（S200 子卡）                            │
│  · 将 beam 草图编码为"稀疏刺激模式"                 │
│  · 模式 = {电极ID, 刺激频率, 刺激序列, 振幅}       │
│  · 编码策略：                                      │
│    - 短词 → 高频短脉冲串                           │
│    - 长词/罕见词 → 间隔式低频脉冲                   │
│    - 语义冲突(beam分歧大) → 多电极异步刺激          │
│    - 语义一致(beam收敛)   → 单区域同步刺激          │
└──────────────────┬──────────────────────────────┘
                   ↓
         刺激信号 → MEA 电极阵列
                   ↓
┌─────────────────────────────────────────────────┐
│ Genius2（湿件）                                    │
│  · 自由演化 ~1–3 秒（不给参考答案，只给刺激）       │
│  · 神经网络在刺激下产生电生理响应                   │
│  · 响应形态取决于：                                │
│    a) 刺激模式（来自 S200 草案）                    │
│    b) 神经元当前状态（受此前数步刺激历史影响）       │
│    c) 培养物个体差异（不同阵列对同一刺激有不同"倾向"） │
└──────────────────┬──────────────────────────────┘
                   ↓
         MEA 读出 → 4096 通道 × 采样时长 的波形矩阵
                   ↓
┌─────────────────────────────────────────────────┐
│ 波形解码器（S200 子卡）                            │
│  · CNN-LSTM 混合解码网络                           │
│  · 输入：4096 通道波形矩阵                          │
│  · 输出：一个"推演修正向量" Δ                        │
│  · Δ 的维度 = 词表大小（256K）                      │
│  · Δ 的每个分量 ∈ [-0.02, 0.02]                    │
│  · ∥Δ∥ 通常 < 0.02（总偏移量小于 2%）               │
└──────────────────┬──────────────────────────────┘
                   ↓
         修正向量 Δ 叠加到 P_silicon 上
                   ↓
┌─────────────────────────────────────────────────┐
│ S200 · 应用修正                                    │
│  · P_final = softmax( log(P_silicon) + α · Δ )    │
│  · α = 1.0（标准模式）/ 0.3（稳定模式）/ 1.8（创意模式）│
│  · 从 P_final 采样 → 当前步最终 Token              │
│  · 注意：采样过程不保证一定偏离 S200 原意            │
└──────────────────┬──────────────────────────────┘
                   ↓
         最终 Token → 拼入输出序列 → 循环至 EOS
```

## 四、刺激模式编码

这是整个架构中唯一一处"跨物种翻译"——将人类语言的符号空间映射到神经元可感知的电刺激空间。

| 编码原则 | 说明 |
|----------|------|
| 不编码语义 | Genius2 没有"词义"。刺激模式不携带语义信息。它携带的是 S200 对当前生成步骤的"困惑度"与"犹豫方向" |
| Beam 分歧编码 | 当 S200 的 top-3 beam 在某个 token 位置上严重分歧时，刺激模式为多电极异步脉冲——多个电极在不同频率上独立运行。Genius2 对此的响应波形在解码后通常表现为较大幅度的 Δ |
| Beam 收敛编码 | 当 top-3 beam 高度一致时，刺激为单电极或同步脉冲。此时的修正向量 Δ 幅值通常很小——S200 本来就有信心的地方，Genius2 不会"多嘴" |
| 上下文记忆 | 刺激模式携带了"距离开头多远"和"当前在回答中的位置"的信息，通过脉冲序列的总体时长编码。Genius2 对回答初期、中期、末期的刺激有不同的基线响应——它在"感觉到"一段对话的节奏 |

> 编码策略的设计原则是一句玩笑但也是实话："我们不告诉 Genius2 人类在说什么。我们只告诉 Genius2，S200 现在有多纠结。"

## 五、Genius2 到底做了什么

Genius2 不是第二个 AI。它是一个由活体神经元构成的、在持续接受 S200 的"情绪刺激"后给出生物电反馈的装置。

我们对它行为的已知事实：

| 已知 | 解释 |
|------|------|
| 修正向量方向在连续 Token 间具有弱相关 | 前一步的刺激影响神经元的突触后电位残余，这种残余持续约 2–3 步 |
| 不同培养阵列对同一刺激给出不同的 Δ | 个体差异。项目组内部给每个培养阵列起了名字——"偏执狂"那个总在否定 S200，"和事佬"那个总是轻微加强 S200 的首选。推理时使用多阵列的均值 Δ，个别阵列的极端值被 τ-clip 削平 |
| 长时间运行后 Δ 幅值趋于衰减 | 神经元疲劳。连续推理三十分钟后，Δ 的有效幅度降至初始的 50%。目前应对策略是分批间歇——每五个 query 暂停 15 秒。研究人员管这叫"让 Genius2 喘口气" |
| 偶尔产生高幅值异常尖峰 | 神经元集群偶尔爆发同步放电，持续时间极短（< 50 ms），强度巨大。对应的解码向量 Δ 在某个极小维度上有一个孤立的大分量（接近 0.02 的边界）。这些"尖峰修正"有时让输出多了一个出人意料的语气词，有时让一个关键动词从"建议"变成了"认为"。目前无人能解释这些尖峰的触发条件 |

> 曾有一次，Opus 被要求用一句话总结《窄门》，S200 草案原定"一个关于爱情与信仰冲突的故事"。Genius2 在"信仰"一词前的两个 Token 处送出一个尖峰修正。最终输出为"一个关于等待自己配得上所爱之人的故事"。
>
> 没有人知道这个尖峰为什么发生。解码器的训练集里没有《窄门》的文本。培养神经元来自一位 23 岁的志愿者——他对《窄门》和纪德一无所知。
>
> 项目组把这个案例记录在案，标签是"巧合"。但每周开组会时，有人会翻到那一页，然后再翻过去。

## 六、推理性能

| 指标 | Opus 2.0 | Sonnet 2.0 | Haiku 2.0 |
|------|----------|------------|-----------|
| S200 推理耗时 | ~18 ms/tok | ~7 ms/tok | ~3 ms/tok |
| 刺激编码耗时 | ~40 ms | ~25 ms | - |
| Genius2 演化耗时 | 1–3 s（可配） | 1–3 s | - |
| 波形解码耗时 | ~60 ms | ~40 ms | - |
| 修正应用耗时 | ~5 ms | ~3 ms | - |
| **总延迟（含 Genius2）** | **~1.2–3.1 s/tok** | **~1.1–3.1 s/tok** | **~3 ms/tok（纯硅基）** |
| 吞吐（含 Genius2，批量=1） | ~0.5 tok/s | ~0.5 tok/s | ~300 tok/s |
| 吞吐（纯 S200 旁路） | ~55 tok/s | ~140 tok/s | ~300 tok/s |

> Haiku 2.0 不使用 Genius2。它太小了，小到 S200 对它毫无"困惑"——beam 几乎总是收敛。没有分歧的刺激模式等于没有信号，Genius2 的响应变成纯噪声。项目组测试过 Haiku+Genius2，结论是"就像一个大学教授对着幼儿园考卷陷入沉思——他沉思不出什么有意义的反馈"。

## 七、Genius2 到底有没有用

这是一个项目组被问过无数次的问题。以下是经过上百组盲测对照的诚实回答：

### 客观差异

- **创意写作**：在开放结尾叙事、诗歌生成、对话创作等场景，双盲评审员选 Opus+Genius2 输出的比例约为 61%（vs 纯 S200 的 39%）。差异在 p<0.01 水平显著。但评审员无法明确说出"为什么选这一篇"——最常出现的评价是"说不清，就是读起来顺一点"或者"这篇感觉像人写的"。评审员不知道自己在和一个硅基草案和一个湿件修正之间的二选一做博弈。
- **多轮对话**：对话超过 15 轮时，Opus+Genius2 在用户满意度上高出约 7 个百分点。主要差异在于避免重复句式——Genius2 似乎对"句式惯性"有微弱的抑制倾向。
- **事实准确性和数学**：无显著差异。Genius2 在需要精确推理的任务上不产生正向贡献，也不产生系统性损害。它只是在那里，偶尔让表达方式偏离 S200 的默认。
- **代码生成**：无显著差异。HBI 计科的学生曾偷偷用 Opus+Genius2 写作业，最终正确率和纯 S200 一致——但变量命名风格更不规律，有个别生成沿用日系命名风格（从英文模式切出 romaji）。究竟是参数量差异还是 Genius2 的干扰，目前无法分辨。

### 主观体验

项目组内部有一个非正式共识：Genius2 给的东西不是"更好的答案"，而是"更有温度的偏差"。它不保证正确，不保证高效，不保证符合任何评估基准上的排名提升。它只保证一件事：当 S200 在两个几乎等概率的措辞之间犹豫时，Genius2 替它做了选择——而且这个选择来自一些活的神经元，不是来自另一层矩阵乘法和 softmax。

> "S200 问 Genius2：'这样说行吗？' Genius2 说：'我不知道你说的什么意思，但刚才那几个字让我觉得……这里偏一点点比较舒服。'"
>
> ——项目组内部对两者关系的非正式描述

## 八、已知问题

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 培养阵列寿命 | 中 | 21 天一个周期。更换时新旧阵列的"行为偏好"不同，输出风格会在更换日后轻微漂移。已有用户注意到"今天 Ripple 好像不太一样"——确实不一样，今天换了一批新神经元 |
| 长序列退化 | 中 | 连续推理超过 30 分钟后修正质量衰减。更严重的是，疲劳的 Genius2 送出的不是趋近零的修正，而是趋近于"随机微小偏移"——一种不可预测的轻微抖动。项目组应对方式是定时强制旁路 Genius2（每30分钟休息5分钟），但用户会注意到响应在这5分钟内"变得有点不一样" |
| 尖峰不可控 | 高 | 无法预测、无法抑制、无法复现。目前唯一的防御是 τ-clip——把任何单个维度的修正强行压到 ±0.02 以内。但这只是防止极端，不是消除。下一次尖峰什么时候来、会撞上哪个词，项目组不知道 |
| 湿件可解释性 | 无 | 没有解释为什么某个刺激导致某个修正。这就是"让活的神经元看着你的草案然后在脑子里自己转一圈再告诉你它觉得哪里不对"的本质——你不能要求一个培养皿里的神经元在输出修正向量之后还给你附一份 NIPS 格式的解释报告 |

## 十、下一步

| 方向 | 状态 |
|------|------|
| 培养阵列寿命延长到 30 天以上 | 进行中。HBI 生科正在测试一种改良培养基配方。挑战是：活得越久，神经元的自发同步放电越强，在"没有刺激时自己在那放电"意味着噪声基底上升 |
| 多阵列差异化挑选 | 已启用但未公开。推理时不是取所有阵列的均值——而是在前几步推理后快速判断哪个阵列"今天的偏好"最适合当前任务类型。相当于在每次对话开始时给 Genius2 阵列群做一个迷你盲测。用到了 HBI 计科 C4 的"潮汐"调度系统的一些调度逻辑 |
| 尖峰溯源 | 停滞。需要更精细的钙成像和单个神经元的 optogenetics 控制——设备不在园区，在当前预算下无法采购 |
| 引入跨培养阵列的"社会学习"范式 | 假说阶段。把两个培养阵列的响应波形相互作为对方的额外输入——让一群神经元看着另一群神经元的反应来做自己的修正。结果完全不可预测。伦理委员会尚未收到此项的正式申请 |

## 附录 A：S200 旁路模式

用户可以在调用时指定 `genius_bypass=true`，此时推理管线跳过 Genius2 全段，退回纯 S200 模式。延迟从 ~1–3 s/tok 降至 ~18 ms/tok（Opus）。

目前 API 上约 4% 的请求使用旁路模式，主要来自代码生成和数学推理场景。但对创意写作请求，旁路模式使用率不到 0.5%。这不是官方引导——是用户自己发现的。
